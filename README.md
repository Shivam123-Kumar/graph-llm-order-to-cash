# 🕸️ Graph LLM: Order-to-Cash Intelligence

[![Streamlit App](https://static.streamlit.io/badges/streamlit_badge_black_white.svg)](https://graph-llm-order-to-cash-yupaaezdxa9pptuhobwbjj.streamlit.app/)

Welcome to the **Graph LLM Order-to-Cash** intelligence system! This project seamlessly blends the analytical capability of Large Language Models (LLMs) with the deeply connected structural representation of a Neo4j Graph Database.

It allows users to ask natural language questions about complex business workflows—specifically the **Order-to-Cash** cycle—and instantly view the data-backed answers and their interconnected network visualized chronologically in an interactive UI.

## ✨ Key Features
- **Natural Language to Cypher:** Utilizes an LLM to accurately translate human questions into complex Neo4j Cypher queries.
- **Interactive Graph Visualization:** Renders the resulting business network dynamically using Streamlit and Pyvis.
- **Intelligent Node Highlighting:** Automatically highlights the specific nodes retrieved by your chat queries, allowing you to instantly pinpoint answers within the broader network.
- **Full Order-to-Cash Modeling:** Captures the absolute operational flow, firmly linking:
  - `Customer` ↔ `Address`
  - `SalesOrder` ↔ `Product`
  - `SalesOrder` → `Delivery` → `Invoice` → `Payment`
  - Financial auditing traces to `JournalEntry`

## 🧠 System Design & Engineering Decisions

### 1. Architecture Decisions
The application uses a decoupled, Python-centric architecture designed for rapid iteration and clear separation of concerns:
* **Frontend (Streamlit):** Chosen for its native Python integration, allowing seamless passing of state (dataframes, graph dictionaries) directly from the backend to the UI without managing complex React hooks or API routes.
* **Visualization (Pyvis):** Chosen because it generates lightweight, interactive, physics-based HTML components that embed securely into Streamlit without browser-crashing overhead.
* **ETL Pipeline (Pandas):** The CSV ingestion is decoupled using standalone orchestration scripts (`preprocess.py`, `load_to_graph.py`) ensuring dirty data is cleansed before it ever touches the graph database.

### 2. Database Choice (Neo4j Aura)
We elected to use Neo4j Aura rather than a traditional relational SQL database (like PostgreSQL) for several core reasons:
* **Join Exhaustion:** An Order-to-Cash trace querying from a `SalesOrder` all the way to a `JournalEntry` requires querying across 5-6 different heavily indexed SQL tables. In Neo4j, these relationships are explicitly stored as edges, making path traversal `$O(1)` instead of an expensive table scan.
* **Schema Evolution:** Graph databases handle disjointed or varying attributes elegantly.
* **LLM Friendliness:** Cypher syntax maps much closer to natural language semantic structures ("Match an Order that was Delivered to a Customer") making the LLM's job generating accurate queries significantly easier than writing nested SQL joins.

### 3. LLM Prompting Strategy
The LLM engine (`cypher_generator.py`) does not operate blindly; it uses a strict **Context-Injected Prompting Strategy**:
* **Explicit Schema Provisioning:** The LLM is provided a strictly defined `GRAPH_SCHEMA` outlining the exact node definitions (`SalesOrder`, `Delivery`, `Invoice`) and explicitly permitted relationships (e.g., `[:DELIVERED_TO]`, `[:GENERATES]`). This prevents the LLM from hallucinating edge names that don't exist.
* **ID Normalization Guidance:** The prompt includes system-level instructions advising the LLM on how IDs are natively stored (preventing type mismatch errors).

### 4. Implementation Guardrails
To harden the system from prompt-injections, infinite loops, and data duplication:
* **Idempotent ETL Loads:** The ingestion scripts exclusively use Neo4j `MERGE` operations instead of `CREATE`. This acts as a database-level guardrail preventing node and edge duplicates even if ingestion scripts are interrupted and re-run.
* **UI Render Caps:** The `graph_builder.py` enforces a hard `LIMIT` fallback on graph paths. If a user asks a broad question ("Show me everything"), the LLM might return 10,000 nodes, which would crash the user's browser via Pyvis. The system safely truncates the render while notifying the user.
* **Read-Only Executor:** The `query_executor.py` operates strictly as a read-layer for LLM-generated output, meaning any accidental `DELETE` or `DROP` Cypher injections generated by the LLM (or malicious user) will be safely restricted.

## 📁 Project Structure
- **`frontend/streamlit_app.py`**: The main chat and graph rendering application.
- **`backend/llm/cypher_generator.py`**: Holds the System Prompt configurations to strictly define the database schema to the LLM context.
- **`backend/graph/graph_builder.py`**: Translates Neo4j query records into Pyvis topology configurations.
- **`scripts/preprocess.py` & `load_to_graph.py`**: The robust ETL toolchain designed to ingest raw line-item datasets, isolate structural key mappings, and deploy the surgical relations into the Neo4j backend.

## 💻 Local Installation

1. Clone the repository and navigate to the project root.
2. Install the required dependencies:
   ```bash
   pip install -r requirements.txt
   ```
3. Create a `.env` file in the root directory and add your cloud credentials:
   ```env
   NEO4J_URI=bolt://<your-aura-instance>
   NEO4J_USER=neo4j
   NEO4J_PASSWORD=<password>
   NEO4J_DATABASE=neo4j
   OPENAI_API_KEY=<your-api-key>
   ```
4. Start the application framework:
   ```bash
   streamlit run frontend/streamlit_app.py
   ```
